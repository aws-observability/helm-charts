# Admission Webhooks

## Purpose
MutatingWebhookConfiguration for the operator. Handles validation and mutation of CRs (AmazonCloudWatchAgent, Instrumentation), pod injection for auto-instrumentation, workload annotation, and namespace monitoring.

## Two TLS Paths
- `operator-webhook.yaml` — Self-signed certs auto-generated by Helm (`autoGenerateCert.enabled: true`). Default path.
- `operator-webhook-with-cert-manager.yaml` — Uses cert-manager for certificate lifecycle. Takes precedence when `certManager.enabled: true`.

Only one renders at a time — they're mutually exclusive via template guards.

## Webhook Types
Both files define the same set of webhooks:
- `instrumentations` — mutate/validate Instrumentation CRs
- `agents` — mutate/validate AmazonCloudWatchAgent CRs
- `pods` — inject auto-instrumentation sidecars (failurePolicy: Ignore by default)
- `workloads` — annotate Deployments/DaemonSets/StatefulSets for monitoring
- `namespaces` — namespace-level monitoring configuration

Each webhook type can be individually enabled/disabled and has its own `failurePolicy` and `namespaceSelector` overrides in `admissionWebhooks`.

## Patterns
- Webhook names follow the `m{resource}.kb.io` / `v{resource}.kb.io` convention (kubebuilder standard)
- The `namePrefix` value prepends to the webhook configuration name for ordering control
- Certificate Secret name is derived via `amazon-cloudwatch-observability.certificateSecretName` helper

## Pitfalls
- Don't set `failurePolicy: Fail` on the pods webhook unless you want pod creation to block when the operator is down.
- The cert-manager variant uses the `cert-manager.io/inject-ca-from` annotation — cert-manager must be installed first.
- `timeoutSeconds` defaults to 10 — increasing it can slow down all mutated resource operations.
